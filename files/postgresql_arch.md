Архитектуру PostgreSQL можно представить как хорошо организованную фабрику, где каждый компонент отвечает за свою часть работы, а взаимодействие между ними построено по принципу клиент-сервер с использованием общей памяти.

Вот подробный разбор основных компонентов и их взаимосвязей.

### 1. Процесс-сервер (Postmaster)

Это «директор завода» — главный управляющий процесс.

- **Запуск и остановка:** Он первым запускается при старте сервера и инициализирует общую память.
    
- **Прием подключений:** Слушает сетевые порты (по умолчанию 5432).
    
- **Порождение потомков:** Когда приходит новый клиент (например, приложение или утилита `psql`), Postmaster создает (форкает) отдельный процесс для обслуживания этого клиента.
    
- **Надзор:** Если дочерний процесс умирает из-за ошибки, Postmaster убивает и перезапускает все «грязные» процессы, чтобы восстановить целостность системы.
    

### 2. Клиентские процессы (Frontend / Backend)

Это «рабочие» на линии.

- **Роль:** Обслуживают конкретное подключение одного клиента.
    
- **Суть:** Каждое соединение `(клиент-приложение)` получает свой отдельный процесс на сервере (в отличие от многопоточных СУБД, где потоки используют общую память более интенсивно, здесь используется многопроцессорность).
    
- **Задачи:**
    
    - Получает SQL-запрос от клиента.
        
    - Проверяет синтаксис и права доступа.
        
    - Строит план выполнения запроса.
        
    - Выполняет запрос, читая данные с диска или из кеша.
        
    - Возвращает результат клиенту.
        

### 3. Общая память (Shared Memory)

Это «доска объявлений» и склад, доступный для всех процессов одновременно.

- **Shared Buffers (Кеш данных):** Самая важная область. Здесь PostgreSQL хранит страницы данных (таблицы, индексы), считанные с диска. Когда процессу нужно прочитать строку, он сначала ищет её здесь. Это значительно ускоряет работу по сравнению с прямым чтением с диска.
    
- **WAL Buffer:** Временное хранилище для журнала предзаписи (Write-Ahead Log). Изменения сначала попадают сюда, а потом сбрасываются на диск.
    
- **CLOG (Commit Log):** Хранит статусы транзакций (выполнена, прервана, в процессе).
    
- **Блокировки (Locks):** Информация о том, кто и какие объекты базы заблокировал.
    

### 4. Вспомогательные фоновые процессы (Background Workers)

Это службы завода, которые не общаются с клиентами напрямую, но обеспечивают жизнедеятельность.

- **Checkpointer (Контрольная точка):** Периодически принудительно сбрасывает все «грязные» (измененные) страницы из _Shared Buffers_ обратно на диск. Это делается для того, чтобы время восстановления после сбоя было фиксированным и небольшим.
    
- **Background Writer (Фоновый писатель):** Постоянно, понемногу записывает «грязные» страницы из _Shared Buffers_ на диск, чтобы сгладить нагрузку и не допустить, чтобы _Checkpointer_ делал всю работу рывками.
    
- **WAL Writer (Запись журнала):** Сбрасывает содержимое _WAL Buffer_ из памяти на диск в файлы WAL (журнал).
    
- **Autovacuum Launcher / Worker (Автовакуум):** Борется с «мертвыми» кортежами (строками), которые остаются после обновлений и удалений (MVCC — механизм многоверсионности). Он чистит мусор и обновляет статистику для планировщика запросов.
    
- **Stats Collector (Сборщик статистики):** Собирает информацию о активности базы: сколько было чтений, вставок, кто к чему обращается.
    
- **Logical Replication Launcher:** Управляет логической репликацией.
    

### 5. Подсистемы и структуры на диске

Это хранилища, где данные живут постоянно, даже когда сервер выключен.

- **Data Directory (PGDATA):** Главная папка с данными.
    
    - **Базовые файлы:** Файлы таблиц и индексов (обычно разбиты на сегменты по 1 ГБ).
        
    - **pg_wal:** Каталог с журналами предзаписи (WAL). Это главный журнал транзакций. Любое изменение сначала записывается сюда, и только потом в таблицы.
        
    - **pg_xact:** Статусы транзакций.
        
    - **postgresql.conf:** Главный файл конфигурации.
        
    - **pg_hba.conf:** Файл, отвечающий за аутентификацию клиентов (кто и откуда может подключаться).
        

### Как они взаимосвязаны? (Схема работы запроса)

1. **Подключение:** Клиент стучится в порт. **Postmaster** принимает запрос, проверяет пароли через `pg_hba.conf` и создает отдельный **клиентский процесс**.
    
2. **Запрос:** Клиент (backend-процесс) получает SQL `UPDATE users SET name = 'John' WHERE id = 1;`.
    
3. **Логирование изменений:** Процесс пишет информацию о грядущем изменении в **WAL Buffer** (в памяти), чтобы в случае сбоя операцию можно было воспроизвести.
    
4. **Поиск данных:** Процесс смотрит в **Shared Buffers**, есть ли там страница с `id=1`.
    
    - Если нет — читает её с диска в Shared Buffers.
        
5. **Блокировка:** Процесс проверяет и ставит блокировку на строку (через механизм блокировок в **общей памяти**), чтобы другие процессы её не испортили.
    
6. **Изменение:** Процесс меняет данные прямо в **Shared Buffers**. Страница теперь помечена как «грязная».
    
7. **Ответ:** Процесс отправляет клиенту `UPDATE 1`.
    
8. **Фоновая работа:**
    
    - **WAL Writer** вскоре сбросит _WAL Buffer_ на диск в `pg_wal`.
        
    - **Background Writer** через какое-то время запишет эту «грязную» страницу из _Shared Buffers_ в основной файл таблицы на диске.
        
    - Если старых версий строк стало много, проснется **Autovacuum** и почистит их.
        
    - Если сервер внезапно упадет сразу после шага 7, при следующем запуске он прочитает **WAL** и восстановит состояние базы на момент завершения транзакции.
        

**Краткое резюме:**  
PostgreSQL — это многопроцессная система, где один главный процесс управляет подключениями, множество рабочих процессов выполняют код клиентов, а фоновые процессы и общая память обеспечивают целостность и производительность.

### Что такое WAL?

**WAL (Write-Ahead Logging)** — это механизм журналирования изменений, который используется в PostgreSQL (и многих других СУБД) для обеспечения надежности и целостности данных. Простыми словами, это **«журнал предзаписи»**: прежде чем изменить данные на диске, система записывает информацию о том, _какие_ изменения планируется сделать, в специальный журнал (WAL). Только после того, как эта запись попала на диск, изменения применяются к самим таблицам и индексам.

Давайте разберем, зачем это нужно, как работает и для чего еще используется WAL в PostgreSQL.

### Зачем нужен WAL?

1. **Гарантия сохранности данных (Durability)**  
    Если в момент записи в таблицу произойдет сбой питания или аварийное отключение сервера, данные в таблице могут остаться в несогласованном состоянии. Благодаря WAL, после перезапуска PostgreSQL может «проиграть» журнал и накатить все изменения, которые были зафиксированы, но не записаны в таблицы. Таким образом, ни одна подтвержденная транзакция не теряется.
    
2. **Снижение накладных расходов на запись**  
    Запись в WAL всегда последовательная (append-only) — новые записи добавляются в конец файла. Это очень быстрая операция, особенно на механических дисках. А запись в таблицы и индексы часто требует случайного доступа к диску (обновление разбросанных страниц). Если бы мы сразу писали в таблицы, производительность была бы ниже. WAL позволяет отложить запись в основные файлы данных, сохраняя при этом надежность.
    
3. **Возможность восстановления на любой момент времени (Point-in-Time Recovery)**  
    Если у вас есть резервная копия базы и все файлы WAL, вы можете восстановить базу данных на любой момент времени в прошлом, вплоть до секунды.
    
4. **Основа для репликации**  
    В PostgreSQL репликация (как физическая, так и логическая) построена на передаче записей WAL с основного сервера на резервные. Реплики применяют эти записи, оставаясь синхронизированными с мастером.
    

---

### Как работает WAL в PostgreSQL?

#### Структура и компоненты

- **WAL-файлы** (сегменты) хранятся в каталоге `pg_wal` внутри директории данных. Они имеют фиксированный размер (обычно 16 МБ) и нумеруются.
    
- **WAL-буфер** — область в общей памяти (shared memory), куда попадают записи о изменениях перед тем, как быть сброшенными на диск.
    
- **WAL-писатель (WAL writer)** — фоновый процесс, который периодически сбрасывает содержимое WAL-буфера на диск, а также обеспечивает синхронизацию при фиксации транзакций.
    

#### Жизненный цикл изменения данных

1. Клиент инициирует транзакцию и выполняет, например, `UPDATE`.
    
2. Соответствующий серверный процесс формирует запись в WAL-буфере: «в таком-то файле таблицы, в такой-то странице, по такому-то смещению будет записано новое значение».
    
3. При фиксации транзакции (`COMMIT`) процесс заставляет WAL-писателя сбросить все накопившиеся записи этой транзакции на диск (гарантированная запись).
    
4. После того как запись попала в WAL-файл на диске, транзакция считается успешно завершенной. Клиент получает подтверждение.
    
5. Изменения в самих таблицах и индексах (в shared buffers) могут быть отложены. Фоновый писатель (background writer) или контрольная точка (checkpoint) позже запишет их на диск в удобное время.
    

#### Восстановление после сбоя

- При запуске PostgreSQL проверяет, были ли применены все изменения из WAL к данным на диске. Если нет, он перечитывает WAL, начиная с последней успешной контрольной точки, и применяет все записи, которые еще не отражены в таблицах. Этот процесс называется **восстановлением (recovery)**.
    

---

### Контрольные точки (Checkpoints)

Чтобы восстановление не длилось бесконечно, PostgreSQL периодически делает **контрольные точки**. Во время контрольной точки все измененные страницы данных, которые находятся в shared buffers, принудительно записываются на диск. После этого специальная запись в WAL отмечает, что все изменения до определенной позиции уже применены к данным. При восстановлении можно начинать чтение WAL именно с этой позиции, игнорируя более старые записи. Это ускоряет перезапуск после сбоя.

---

### Режимы синхронизации WAL

Поведение WAL при фиксации транзакции можно настроить через параметр `synchronous_commit`:

- **on** (по умолчанию) — возврат управления клиенту происходит только после того, как WAL-запись о транзакции попала на диск (надежно, но медленнее).
    
- **off** — подтверждение возвращается сразу, без ожидания записи WAL на диск (быстрее, но есть риск потери последних транзакций при сбое).
    
- **remote_write / remote_apply** — используются в синхронной репликации для дополнительной гарантии.
    

---

### Роль WAL в репликации

- **Физическая репликация (streaming replication)**: Реплика подключается к мастеру и непрерывно получает копии WAL-записей, применяя их к своей копии данных. Так реплика всегда отстает от мастера не более чем на несколько транзакций.
    
- **Логическая репликация**: Специальный модуль декодирует WAL и восстанавливает логические операции (вставки, обновления, удаления), которые можно применять на других базах (даже с другой схемой или на другой версии PostgreSQL).
    

---

### Итог

**WAL** — это «черный ящик» PostgreSQL, который записывает каждое изменение еще до того, как оно коснется основных файлов данных. Без него СУБД не могла бы гарантировать сохранность данных при сбоях, а такие возможности, как репликация и восстановление на момент времени, были бы невозможны. WAL — фундаментальный компонент, обеспечивающий **надежность, производительность и масштабирование** PostgreSQL.


### Что такое MVCC?

**MVCC (Multiversion Concurrency Control)** — это механизм управления конкурентным доступом к данным, используемый в PostgreSQL (и многих других современных СУБД). Он позволяет нескольким транзакциям одновременно работать с базой данных, не блокируя друг друга при чтении, и гарантирует согласованность данных с точки зрения каждой транзакции.

Простыми словами, MVCC можно сравнить с **системой контроля версий для строк таблицы**. Вместо того чтобы перезаписывать данные, PostgreSQL сохраняет несколько версий одной и той же строки, помечая, для каких транзакций какая версия актуальна.

---

### Зачем нужен MVCC?

Основная цель — обеспечить свойства **ACID** (атомарность, согласованность, изоляция, долговечность) без использования тяжёлых блокировок. В частности, MVCC реализует уровень изоляции **Snapshot Isolation** (мгновенный снимок), где каждая транзакция видит согласованный снимок базы данных на момент своего начала, независимо от изменений, вносимых другими транзакциями.

Благодаря MVCC:

- **Читатели не блокируют писателей** — транзакция, читающая данные, не мешает другой транзакции изменять те же строки.
    
- **Писатели не блокируют читателей** — пока одна транзакция обновляет строку, другие могут читать её старую версию.
    
- **Каждая транзакция работает со своим согласованным снимком**, что исключает «грязное» чтение, неповторяющееся чтение и фантомы (в зависимости от уровня изоляции).
    

---

### Как MVCC реализован в PostgreSQL?

В PostgreSQL каждая строка таблицы хранит скрытую служебную информацию, которая определяет её видимость для транзакций. Основные элементы:

- **xmin** — идентификатор транзакции, создавшей данную версию строки.
    
- **xmax** — идентификатор транзакции, удалившей или изменившей данную версию (если строка удалена или обновлена, xmax указывает на транзакцию, которая это сделала).
    
- **Другие системные поля** (cmin, cmax) для отслеживания команд внутри транзакции, но для понимания MVCC достаточно xmin и xmax.
    

Идентификаторы транзакций (XID) — это 32-битные счётчики, которые монотонно возрастают. Каждая новая транзакция получает следующий XID.

#### Примеры операций

**INSERT**  
При вставке новой строки:

- Устанавливается `xmin = XID_текущей_транзакции`.
    
- `xmax = 0` (означает, что строка не удалена и не изменена).
    

**DELETE**  
При удалении строки:

- Текущая транзакция устанавливает `xmax = XID_текущей_транзакции` для этой версии строки.
    
- Сама строка физически остаётся в таблице, но помечается как «мёртвая» для всех последующих транзакций, которые увидят, что xmax соответствует завершённой транзакции удаления.
    

**UPDATE**  
В PostgreSQL UPDATE выполняется как DELETE + INSERT:

- Старая версия строки помечается `xmax = XID_текущей_транзакции` (как удалённая).
    
- Создаётся новая версия строки с `xmin = XID_текущей_транзакции` и новыми данными.
    

#### Определение видимости

Когда транзакция читает данные, она просматривает версии строк и решает, видит ли она конкретную версию, на основе правил:

- Версия видна, если `xmin` уже зафиксирован (committed) и меньше или равен XID текущей транзакции (или находится в специальном диапазоне «замороженных» строк).
    
- Версия не видна, если `xmax` уже зафиксирован и указывает на удаление до начала текущей транзакции.
    

Для этого PostgreSQL ведёт карту завершённых транзакций (commit log — CLOG), где хранятся статусы (в процессе, зафиксирована, откачена).

---

### Аналогия для понимания

Представьте таблицу как книгу учёта, где каждая запись ведётся карандашом, но изменения не стираются, а дописываются новые версии, а старые остаются, но помечаются как недействительные для определённых читателей.

**Пример**:

1. Транзакция А (XID=100) читает строку с балансом = 1000.
    
2. Транзакция Б (XID=101) меняет баланс на 900: старая версия (1000) помечается `xmax=101`, новая версия (900) создаётся с `xmin=101`.
    
3. Транзакция А продолжает читать — она видит старую версию (1000), потому что новая версия создана позже начала А (101 > 100) и ещё не зафиксирована.
    
4. Когда транзакция Б фиксируется, все последующие транзакции (с XID >= 102) увидят новую версию (900), если не начались раньше.
    

---

### Преимущества MVCC

- **Высокая конкурентность** — чтения никогда не блокируются записями, записи не блокируют чтения.
    
- **Согласованность без блокировок** — каждая транзакция получает целостный снимок базы на момент её начала.
    
- **Гибкость уровней изоляции** — можно реализовать Read Committed, Repeatable Read, Serializable поверх snapshot isolation.
    

---

### Недостатки и плата за MVCC

- **Хранение старых версий** — «мёртвые» строки занимают место (bloat), пока не будут очищены.
    
- **Накладные расходы на обслуживание** — нужен механизм очистки (VACUUM), который удаляет версии, уже не нужные ни одной активной транзакции.
    
- **Дополнительные поля в строках** — служебная информация увеличивает размер строки.
    
- **Риск переполнения XID** — поскольку XID — 32-битное число (максимум ~4 миллиарда), требуется периодическая «заморозка» старых строк, чтобы они считались видимыми для всех транзакций (этим тоже занимается VACUUM).
    

---

### Связь с VACUUM и Autovacuum

Именно из-за MVCC в PostgreSQL необходим **VACUUM**. Он:

- Удаляет мёртвые версии строк (помеченные xmax завершёнными транзакциями).
    
- Освобождает место внутри страниц.
    
- «Замораживает» старые строки (устанавливает им специальную метку, чтобы избежать переполнения XID).
    

**Autovacuum** автоматизирует этот процесс, запуская очистку по мере накопления мёртвых строк.

---

### Итог

**MVCC** — это фундаментальный принцип работы PostgreSQL, который обеспечивает:

- Изоляцию транзакций без блокировок.
    
- Мгновенные согласованные снимки данных.
    
- Высокую производительность при параллельной работе.
    

Расплата за эти возможности — необходимость регулярной очистки старых версий, что реализуется механизмами VACUUM и Autovacuum. Понимание MVCC помогает правильно проектировать базы данных, настраивать обслуживание и диагностировать проблемы с производительностью.

---
### Зачем нужен Vacuum и Autovacuum?

**VACUUM** и **Autovacuum** — это механизмы PostgreSQL для обслуживания таблиц и борьбы с так называемым «раздуванием» (bloat), возникающим из-за особенностей реализации многоверсионности (MVCC). Давайте разберемся, зачем они нужны, как работают и чем отличаются.

---

## 1. Почему нужна очистка? (Проблема MVCC)

В PostgreSQL при выполнении команд `UPDATE` или `DELETE` старые версии строк физически не удаляются из таблицы сразу. Вместо этого:

- **UPDATE** помечает старую версию строки как «мертвую» и создает новую версию этой же строки.
    
- **DELETE** просто помечает строку как «мертвую».
    

Со временем количество «мертвых» (неактуальных) версий строк накапливается. Это приводит к:

- **Раздуванию таблиц и индексов** – место на диске занимается, но полезной информации там мало.
    
- **Снижению производительности** – при сканировании таблицы приходится просматривать много ненужных строк.
    
- **Риску исчерпания идентификаторов транзакций** (transaction ID wraparound) – если вовремя не «заморозить» старые строки, счётчик транзакций может переполниться (это аварийная ситуация).
    

Именно для решения этих проблем и нужны VACUUM и Autovacuum.

---

## 2. VACUUM (ручная очистка)

**VACUUM** — это команда (и утилита), которую администратор может запустить вручную для обслуживания конкретной таблицы или всей базы данных.

### Основные действия VACUUM:

- **Удаление «мертвых» версий строк** — освобождает место внутри страниц данных, помечая его как доступное для будущих вставок.
    
- **Обновление статистики** (если добавлен параметр `ANALYZE`) — помогает планировщику запросов выбирать оптимальные планы.
    
- **Защита от переполнения счётчика транзакций** — «заморозка» старых строк (установка специальной метки, что строка актуальна для всех текущих и будущих транзакций).
    
- **Освобождение места в индексах** — очищает ссылки на удалённые строки.
    

### Виды ручного VACUUM:

- **Обычный VACUUM**  
    Запускается без блокировки таблицы (чтение и запись разрешены). Он очищает «мёртвые» кортежи, но не возвращает занятое место операционной системе — место остаётся зарезервированным за таблицей и может быть использовано для новых вставок.
    
        VACUUM my_table;
    
- **VACUUM FULL**  
    Полная перестройка таблицы и индексов. Таблица блокируется (эксклюзивная блокировка), но место возвращается ОС, и таблица сжимается до минимального размера. Используется редко, когда нужно реально уменьшить занимаемое место.
    
	    VACUUM FULL my_table;
    
- **VACUUM ANALYZE**  
    Одновременно выполняет очистку и обновление статистики.
    
	    VACUUM ANALYZE my_table;
    

---

## 3. Autovacuum (автоматическая очистка)

**Autovacuum** — это фоновый процесс PostgreSQL, который автоматически запускает операции VACUUM (и при необходимости ANALYZE) для таблиц, когда это необходимо. Он включён по умолчанию и избавляет администратора от ручного вмешательства.

### Как работает Autovacuum?

1. **Сбор статистики**  
    Процесс `stats collector` непрерывно собирает информацию о количестве вставок, обновлений и удалений в каждой таблице.
    
2. **Пороги срабатывания**  
    Для каждой таблицы вычисляется, сколько «мертвых» кортежей в ней накопилось. Если это число превышает заданный порог, Autovacuum запускает очистку.  
    Порог вычисляется по формуле:  
    `порог = autovacuum_vacuum_threshold + autovacuum_vacuum_scale_factor * число_живых_кортежей`  
    По умолчанию `threshold = 50`, `scale_factor = 0.2` (т.е. очистка начнётся, когда «мертвых» строк станет больше 50 + 20% от размера таблицы).
    
3. **Запуск рабочих процессов**  
    Если порог превышен, главный процесс `autovacuum launcher` порождает рабочий процесс `autovacuum worker`, который подключается к конкретной таблице и выполняет очистку (обычный VACUUM, без блокировки). Для разных таблиц могут одновременно работать несколько воркеров (максимум задаётся параметром `autovacuum_max_workers`).
    
4. **Дополнительно: autovacuum analyze**  
    Аналогично считается порог для запуска автоматического ANALYZE (обновление статистики).
    

### Важные настройки (в postgresql.conf):

- `autovacuum = on` — включает/выключает.
    
- `autovacuum_naptime = 1min` — интервал проверки необходимости очистки.
    
- `autovacuum_vacuum_threshold` и `autovacuum_vacuum_scale_factor` — глобальные пороги.
    
- `autovacuum_freeze_max_age` — максимальный возраст транзакций до принудительной заморозки (предотвращает wraparound).
    
- Для каждой таблицы можно переопределить параметры через хранение (`ALTER TABLE ... SET`).
    

---

## 4. Разница между VACUUM и Autovacuum

|Характеристика|VACUUM (ручной)|Autovacuum|
|---|---|---|
|**Инициатор**|Администратор (вручную или по cron)|Автоматически, фоновым процессом|
|**Регулярность**|По необходимости|Постоянно, в зависимости от активности|
|**Управление**|Полный контроль над моментом запуска|Настраивается через конфигурацию|
|**Блокировки**|Обычный VACUUM не блокирует, VACUUM FULL блокирует|Всегда обычный VACUUM (без блокировок)|
|**Зачем нужен**|Срочная очистка, уменьшение файлов (FULL), разовые задачи|Поддержание таблиц в чистоте в повседневной работе|

Обычно в современных PostgreSQL достаточно полагаться на Autovacuum. Ручной VACUUM требуется в особых случаях: после массовых загрузок/удалений, при настройке нестандартных таблиц, или когда нужно выполнить VACUUM FULL.

---

## 5. Почему важно следить за работой Autovacuum?

Если Autovacuum не успевает или отключён, возникают проблемы:

- **Разрастание таблиц (bloat)** — запросы замедляются, дисковое пространство расходуется впустую.
    
- **Замедление индексов** — индексы содержат ссылки на «мертвые» строки.
    
- **Transaction ID wraparound** — самая опасная ситуация. Когда возраст самых старых незамороженных кортежей приближается к 2 миллиардам, PostgreSQL начинает выдавать предупреждения и в конце концов переходит в режим «только чтение» или требует немедленного VACUUM. Autovacuum как раз предотвращает это, своевременно замораживая старые строки.
    

Поэтому мониторинг активности autovacuum (логи, pg_stat_* таблицы) — важная часть администрирования.

---

## 6. Примеры мониторинга

``` sql
-- Посмотреть, сколько «мертвых» кортежей в таблицах:
SELECT relname, n_dead_tup, last_vacuum, last_autovacuum
FROM pg_stat_user_tables;

-- Узнать, активен ли autovacuum сейчас:
SELECT * FROM pg_stat_activity WHERE query LIKE 'autovacuum:%';

-- Настроить агрессивный autovacuum для конкретной таблицы (например, для часто обновляемой):
ALTER TABLE my_table SET (autovacuum_vacuum_scale_factor = 0.05);
```

### Резюме

- **VACUUM** — инструмент для очистки «мертвых» строк вручную.
    
- **Autovacuum** — фоновый демон, автоматически делающий ту же работу по заданным правилам.
    
- Оба механизма критически важны для поддержания производительности и стабильности PostgreSQL. Без них база данных быстро деградирует.
