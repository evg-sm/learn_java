
⏺ Аргумент компилятора -Xjsr305=strict в Kotlin управляет обработкой аннотациий nullability из JSR-305.                                                                   

### JSR-305 и nullability 

JSR-305 — это стандарт Java с аннотациями для определения null-safety:  
- @NonNull — значение не может быть null
- @Nullable — значение может быть null  
- @CheckForNull — проверь на null перед использованием

### Режимы -Xjsr305 

-Xjsr305={strict|warn|ignore|default}                                                                                                       
- strict - Самый строгий режим — Kotlin treats JSR-305 nullability annotations as hard errors:                                                                                     
```java
// Java код с @NonNull
public @NonNull String getName() { ... }

// Kotlin видит это как:
fun getName(): String // non-null

// Если вернёт null — compile error, а не warning
val name: String = javaObj.getName() // гарантированно non-null 
```

#### warn (default)                                                                                                                                                          
Компилятор выдаёт предупреждения, но код компилируется:                                                    
```java
// Предупреждение о потенциальном null            
val name: String = javaObj.getName() // warning, но компилируется                
```

#### ignore                                                                                                                                                            
Полностью игнорирует JSR-305 аннотации — все типы nullable из Java.                                     		  
#### default                                                                                                                                         

Аналогичен warn для большинства аннотаций, но strict для:
 - @NonNull из конкретных библиотек (например, Jetbrains annotations)

#### Когда использовать strict
Рекомендуется для новых проектов или при миграции на null-safety: 
```kotlin
// build.gradle.kts   
kotlin {
  compilerOptions {
    freeCompilerArgs.add("-Xjsr305=strict")
  }
}
```

### Преимущества:  

- Строгая null-safety граница Java/Kotlin
- Меньше NPE в runtime                                                                                                                     
- Явные контракты API 														


### Риски: 

- Может сломать существующий код                                      
- Требует аккуратной работы с Java-библиотеками 																						  
Пример в Gradle 

```kotlin
// build.gradle.kts   
tasks.withType<KotlinCompile> {
  kotlinOptions {
	  freeCompilerArgs = listOf("-Xjsr305=strict")
  }       
}
```
