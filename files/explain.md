
Инструкция ***EXPLAIN*** в PostgreSQL используется для **отображения плана выполнения запроса**, который выберет ***оптимизатор запросов***. Она не выполняет сам запрос, а только анализирует и выводит, как PostgreSQL планирует его выполнить — это полезно для **оптимизации производительности**.

### Основные функции `EXPLAIN`:

- Показывает **последовательность операций** (например, сканирование таблицы (scan), соединения (join), сортировки (sort), фильтрации (filter)).
- Указывает **оценку стоимости** выполнения каждой операции (в условных единицах).
- Показывает **ожидаемое количество строк** (rows) и **ширину строк** (width).
- Помогает выявить **узкие места**: полные сканирования таблиц (Seq Scan), 
  не используемые индексы, дорогие соединения и т.д.

### Полезные варианты:

- **`EXPLAIN ANALYZE`** — выполняет запрос и показывает **реальные** временные затраты и количество строк (в отличие от оценок).
- **`EXPLAIN (ANALYZE, BUFFERS)`** — выполняет запрос и показывает **реальные** временные затраты и количество строк (в отличие от оценок). ***BUFFERS*** - Показывает **количество операций чтения и записи буферов** — это ключевой показатель для понимания, насколько запрос эффективно использует оперативную память (shared buffers) и сколько пришлось обратиться к диску.
- **`EXPLAIN (VERBOSE, COSTS OFF)`** — выводит более детальную информацию, без оценок стоимости (для читаемости).
- **`EXPLAIN (FORMAT JSON)`** — выводит план в формате JSON, удобно для автоматической обработки.

### Типы scan:
#### 1. **Seq Scan (Sequential Scan) — Последовательное сканирование**

- **Что делает**: Читает **все строки таблицы** по порядку, от начала до конца.
- **Когда используется**:
    - Нет подходящего индекса.
    - Запрос возвращает **большую часть таблицы** (например, > 10–20% строк).
    - Таблица очень маленькая.
- **Плюсы**: Просто, надёжно, не требует индексов.
- **Минусы**: **Медленно** на больших таблицах — может читать гигабайты данных.

#### 2. **Index Scan — Сканирование по индексу**

- **Что делает**: Использует **B-дерево индекс** (обычный `INDEX`) для поиска **конкретных строк**, затем обращается к таблице (heap) за остальными полями.
- **Когда используется**:
    - Есть индекс на поле в `WHERE`, `JOIN`, `ORDER BY`.
    - Запрос возвращает **мало строк**.
- **Особенность**: Для каждой найденной строки делает **дополнительное обращение к таблице** (random I/O).
#### 3. **Index Only Scan — Сканирование только по индексу**

- **Что делает**: **Не обращается к таблице** — все нужные данные есть **прямо в индексе**.
- **Когда используется**:
    - Все столбцы в `SELECT`, `WHERE`, `ORDER BY` входят в индекс.
    - Индекс **покрывающий (covering)**.
    - Строки не были изменены после последнего `VACUUM` (см. `visibility map`).
- **Плюсы**: **Самый быстрый тип сканирования** — нет обращения к таблице!
- **Минусы**: Требует **специально созданного индекса**.


### Пример (Основные элементы вывода):

  Sort  (cost=100..110 rows=1000) (actual time=0.5..0.8 ms)
    Buffers: temp read=5 written=10
    -> Seq Scan on orders  (actual time=0.01..0.3 ms)
         Buffers: shared hit=50 read=200

#### 1. **План выполнения**
Каждый узел показывает операцию (Scan, Join, Sort и т.д.) с метриками:
Seq Scan on users  (cost=0.00..35.50 rows=2550 width=128) (actual time=0.015..0.089 ms)
- cost — оценка планировщика (startup..total)
- actual time — реальное время (first row..last row)

#### 2. **BUFFERS** — ключевые метрики

Buffers: shared hit=120 read=5
Buffers: temp read=10 written=20
Buffers: local hit=50

  Типы буферов:
  - shared — буферы из общей памяти (таблицы, индексы)
  - temp — временные файлы на диске (сортировки, хеши)
  - local — локальные буферы (временные таблицы сессии)

  Операции:
  - hit — найдено в памяти (быстро)
  - read — прочитано с диска (медленно, I/O)
  - written — записано на диск
  - dirtied — страница изменена в памяти

#### 3. **Что искать**

| Проблема        | Индикатор                                |
| --------------- | ---------------------------------------- |
| Нехватка памяти | *temp read/written > 0*                  |
| Missing индексы | *shared read >> shared hit при Seq Scan* |
| Эффективный кэш | *shared hit доминирует*                  |

 #### 4. **Пример анализа**

  Sort  (cost=100..110 rows=1000) (actual time=0.5..0.8 ms)
    Buffers: temp read=5 written=10
    -> Seq Scan on orders  (actual time=0.01..0.3 ms)
         Buffers: shared hit=50 read=200

  Вывод:
  - Сортировка не влезла в work_mem (temp I/O)
  - Таблица не в кэше (200 read) — первый запуск или маленький shared_buffers
#### ***5. Практические советы***

- Сравнивайте actual vs cost — большие расхождения = устаревшая статистика (ANALYZE)
- Минимизируйте temp — увеличьте work_mem
- Мониторьте shared read — частые чтения = нужен больший кэш или индексы
-  Запускайте несколько раз — первый вызов холодный (без кэша)
