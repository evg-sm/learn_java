## Java 21: Виртуальные потоки

До появления виртуальных потоков в Java было два основных подхода к разработке:
- Блокирующий подход
- Асинхронный подход
## Блокирующий подход
Классика программирования. Spring MVC.
***Концепция: 1 запрос – 1 поток***

```kotlin
private fun selectUserFromDataBase(userId: Long): User {  
    val dataBaseRequest = createRequest(userId)  
    val rowSet = dataBaseClient.performRequest(dataBaseRequest)  
    val user = remapUser(rowSet)  
    return user  
}
```

В таблице ниже приведен график времени и утилизации CPU

| action                                         | time                                       | cpu                                   |
| ---------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| createRequest(userId)                          | 0.000100 ms                                | CPU 100%                              |
| dataBaseClient.performRequest(dataBaseRequest) | <font style="color:red">100.0000 ms</font> | <font style="color:red">CPU 0%</font> |
| remapUser(rowSet)                              | 0.000100 ms                                | CPU 100%                              |
***Основная проблема - процессор простаивает 99% времени на операциях I/O***

## Основаная задача:
- Увеличить утилизацию процессора.
- Увеличить пропускную способность приложения.

И тут на сцену выходит

## Асинхронный подход

Реактивный стек, полностью асинхронный подход к программированию.
***Концепция: несколько запросов – 1 поток***
#### Плюсы:
- Улучшают пропускную способность приложения
- Эффективнее утилизирует CPU
#### Минусы:
- Много фреймворков: RxJava, Reactor, WebFlux, CompletableFuture
- Высокий порог входа
- Сложно читаемый код
- Сложности с отладкой
- Сложности с обработкой ошибок
- Сложности с тестированием

## Решение:  Создавать больше потоков

Почему такой подход не работает c платформенными потоками Java:
- Потоки в Java - это обертка над потоками ядра (потоками платформы)
- Создание нового потока требует ~2 MB оперативной памяти, создание, например, миллиона потоков потребует 2 TB оперативной памяти
- Создание нового потока требует времени, порядка нескольких миллисекунд, для создания миллиона потоков потребуется тысяча секунд(~15 минут)
- Если попробуем создавать потоки таким образом, то упремся в лимиты системы задолго до того, как создадим миллион потоков (1000-2000 потоков, в зависимости от железа)

## Решение:  Виртуальные потоки Java

***Концепция: 1 запрос – 1 поток***

### Преимущества виртуальных потоков:
- Виртуальные потоки - это новые потоки в мире Java
- Виртуальные потоки – легковесные, они недороги для создания
- Можно создать миллион потоков
-  Улучшают пропускную способность приложения
- Снижается потребление памяти приложением
- Больше не надо опираться на асинхронный неблокирующий код (асинхронный подход)
- Не нужно создавать пулы потоков, виртуальные потоки создаются по запросу - новый на каждый запрос, после чего уничтожаются
- Идеально подходят для блокирующего кода, такой как вызов REST, обращение к БД и т.д.
- Эффективнее утилизирует CPU

## Виртуальные потоки в Java

### Ключевые моменты

**Виртуальные потоки (Virtual Threads)** - это легковесные потоки, представленные в Java 21 как часть Project Loom. Основные характеристики:
- **Легковесность**: Потребляют мало памяти (начальный размер ~200-300 байт)
- **Масштабируемость**: Можно создавать миллионы виртуальных потоков
- **Управление JVM**: Планируются и управляются виртуальной машиной Java
- **Совместимость**: Используют тот же API, что и обычные потоки (`java.lang.Thread`)

### Отличия от физических потоков

| Аспект               | Виртуальные потоки               | Физические потоки (Platform Threads) |
| -------------------- | -------------------------------- | ------------------------------------ |
| **Реализация**       | Реализованы в JVM                | Обертки вокруг потоков ОС            |
| **Память**           | ~200-300 байт                    | ~1-2 МБ (стек по умолчанию)          |
| **Создание**         | Быстрое (наносекунды)            | Медленное (миллисекунды)             |
| **Масштабируемость** | Миллионы потоков                 | Тысячи потоков                       |
| **Планирование**     | JVM планирует на carrier-потоках | ОС планирует на ядрах CPU            |
| **Блокировка**       | Дешевая, не блокирует OS-поток   | Дорогая, блокирует OS-поток          |
### Преимущества виртуальных потоков
1. **Высокая производительность I/O операций**
2. **Упрощение конкурентного кода**    
3. **Эффективное использование ресурсов**
  - Один физический поток может обслуживать множество виртуальных потоков
  - Виртуальные потоки автоматически освобождаются при блокировке

### Потенциальные проблемы
### ThreadLocal загрязнение

**ThreadLocal загрязнение (ThreadLocal pollution)** — это ситуация, когда данные, сохранённые в `ThreadLocal` переменной одним виртуальным потоком, остаются в carrier-потоке (физическом потоке-носителе) после завершения виртуального потока. Затем другой виртуальный поток, выполняющийся на том же carrier-потоке, может некорректно увидеть эти "чужие" данные.
### Причина

Виртуальные потоки не привязаны постоянно к одному физическому потоку. Они выполняются на carrier-потоках из общего пула (например, `ForkJoinPool`). Когда виртуальный поток блокируется на I/O или другой операции, он освобождает carrier-поток, который может быть переиспользован для другого виртуального потока.

`ThreadLocal` переменные хранятся в самом потоке (в поле `ThreadLocalMap`). Для обычных потоков это логично — данные живут вместе с потоком. Но для виртуальных потоков проблема в том, что:

- Виртуальный поток устанавливает значение в `ThreadLocal`.
    
- Затем он завершается или приостанавливается, но не очищает `ThreadLocal`.
    
- Carrier-поток, на котором выполнялся виртуальный поток, сохраняет это значение в своей `ThreadLocalMap`.
    
- Следующий виртуальный поток, выполняемый на этом же carrier, может прочитать это "левое" значение, если обратится к тому же `ThreadLocal`.
    

Это и есть **загрязнение** — данные одного виртуального потока "просачиваются" в другой.

## Последствия

- **Утечка контекста**: например, идентификатор пользователя или транзакции может случайно использоваться в другом потоке, что приведёт к нарушению безопасности или логическим ошибкам.
    
- **Некорректное поведение**: код, полагающийся на `ThreadLocal` для хранения состояния (например, текущая сессия, соединение с БД), может работать непредсказуемо.
    
- **Трудности отладки**: ошибки проявляются нерегулярно, в зависимости от того, как виртуальные потоки распределяются по carrier-потокам.
