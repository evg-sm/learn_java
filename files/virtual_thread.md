## ***Java 21: Виртуальные потоки***

До появления виртуальных потоков в Java было два основных подхода к разработке:
- Блокирующий подход
- Асинхронный подход
У обоих подходов есть свои недостатки
## ***Блокирующий подход***

Классика программирования. Spring MVC.
***Концепция: 1 запрос – 1 поток***

```kotlin
private fun selectUserFromDataBase(userId: Long): User {  
    val dataBaseRequest = createRequest(userId)  
    val rowSet = dataBaseClient.performRequest(dataBaseRequest)  
    val user = remapUser(rowSet)  
    return user  
}
```

В таблице ниже приведен график времени и утилизации CPU

| action                                         | time                                       | cpu                                   |
| ---------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| createRequest(userId)                          | 0.000100 ms                                | CPU 100%                              |
| dataBaseClient.performRequest(dataBaseRequest) | <font style="color:red">100.0000 ms</font> | <font style="color:red">CPU 0%</font> |
| remapUser(rowSet)                              | 0.000100 ms                                | CPU 100%                              |
<font style="color:red">Основная проблема - процессор простаивает 99% времени на операциях I/O</font>
## Основная задача:
- Увеличить утилизацию процессора.
- Увеличить пропускную способность приложения.

И тут на сцену выходит "***Асинхронный подход***"

## ***Асинхронный подход***

Реактивный стек, полностью асинхронный подход к программированию.
***Концепция: несколько запросов – 1 поток***
#### Плюсы:
- Улучшают пропускную способность приложения
- Эффективнее утилизирует CPU
#### Минусы:
- Много фреймворков: RxJava, Reactor, WebFlux, CompletableFuture
- Высокий порог входа
- Сложно читаемый код
- Сложности с отладкой
- Сложности с обработкой ошибок
- Сложности с тестированием

---
## ***Решение:  Создавать больше потоков***

Почему такой подход ***не работает c платформенными потоками Java***:
- Потоки в Java - это обертка над потоками ядра (потоками платформы)
- Создание нового потока требует ~2 MB оперативной памяти, создание, например, миллиона потоков потребует 2 TB оперативной памяти
- Создание нового потока требует времени, порядка нескольких миллисекунд, для создания миллиона потоков потребуется тысяча секунд(~15 минут)
- Если попробуем создавать потоки таким образом, то упремся в лимиты системы задолго до того, как создадим миллион потоков (1000-2000 потоков, в зависимости от железа)

## ***Виртуальные потоки в Java***

### Ключевые моменты

**Виртуальные потоки (Virtual Threads)** - это легковесные потоки, представленные в Java 21 как часть Project Loom. Основные характеристики:
- **Легковесность**: Потребляют мало памяти (начальный размер ~200-300 байт)
- **Масштабируемость**: Можно создавать миллионы виртуальных потоков
- **Управление JVM**: Планируются и управляются виртуальной машиной Java
- **Совместимость**: Используют тот же API, что и обычные потоки (`java.lang.Thread`)

### Отличия от физических потоков

| Аспект               | Виртуальные потоки               | Физические потоки (Platform Threads) |
| -------------------- | -------------------------------- | ------------------------------------ |
| **Реализация**       | Реализованы в JVM                | Обертки вокруг потоков ОС            |
| **Память**           | ~200-300 байт                    | ~1-2 МБ (стек по умолчанию)          |
| **Создание**         | Быстрое (наносекунды)            | Медленное (миллисекунды)             |
| **Масштабируемость** | Миллионы потоков                 | Тысячи потоков                       |
| **Планирование**     | JVM планирует на carrier-потоках | ОС планирует на ядрах CPU            |
| **Блокировка**       | Дешевая, не блокирует OS-поток   | Дорогая, блокирует OS-поток          |

## ***Устройство виртуальных потоков в Java***

### Архитектура виртуальных потоков

#### Базовые компоненты
1. **Virtual Thread** - легковесный поток, управляемый JVM
2. **Carrier Thread** - физический поток ОС, который выполняет код виртуального потока
3. **Scheduler** - планировщик в JVM, который назначает виртуальные потоки на carrier-потоки
4. **Continuation** - основная единица выполнения, которая может приостанавливаться и возобновляться

### Что такое Carrier Thread?

**Carrier Thread** - это обычный физический поток платформы (platform thread), который выступает в роли "носителя" для выполнения виртуальных потоков.
```java
// Carrier thread работает в thread pool'е ForkJoinPool
ForkJoinPool carrierPool = ForkJoinPool.commonPool();

// Каждый виртуальный поток выполняется на каком-то carrier-потоке
Thread virtualThread = Thread.startVirtualThread(() -> {
    // Этот код выполняется на carrier-потоке
    System.out.println("Virtual thread running on carrier: " + 
        Thread.currentThread().toString());
});
```

### Состояния виртуального потока
```java
// Виртуальный поток имеет те же состояния, что и обычный
Thread virtualThread = Thread.ofVirtual()
    .name("vt-", 0)
    .unstarted(() -> {
        // NEW -> RUNNABLE
        System.out.println("Starting...");
        
        // При блокировке: RUNNABLE -> WAITING
        LockSupport.park();
        
        // При I/O: RUNNABLE -> BLOCKED (виртуально)
        HttpClient.newHttpClient().send(request, BodyHandlers.ofString());
    });
```
### Механизм Continuation

```java
// Концептуально виртуальный поток работает так:
class VirtualThreadContinuation {
    private Continuation cont;
    
    void run() {
        while (!cont.isDone()) {
            // При блокировке I/O continuation приостанавливается
            cont = suspendAtBlockingOperation();
            
            // Carrier поток освобождается для других виртуальных потоков
            carrierThread.yieldToOtherVirtualThread();
        }
    }
}
```

### Процесс планирования

```java
Виртуальный поток [VT1] ---блокировка I/O---> [Освобождение carrier]
                                     ↓
Carrier поток [C1] ---переключение---> [Выполнение VT2]
                                     ↓
Когда I/O завершен ---продолжение---> [VT1 возобновляется на любом carrier]
```

## Преимущества такой архитектуры

1. **Эффективность**: Carrier потоки используются максимально эффективно
2. **Масштабируемость**: Миллионы виртуальных потоков на ограниченном числе carrier'ов
3. **Совместимость**: Существующий код работает без изменений
4. **Гибкость**: Можно настраивать планировщик под конкретные нужды

***Такая архитектура позволяет достичь высокой производительности I/O операций, сохраняя при этом простоту синхронного стиля программирования.***

### ***Потенциальные проблемы***
### ThreadLocal загрязнение

**ThreadLocal загрязнение (ThreadLocal pollution)** — это ситуация, когда данные, сохранённые в `ThreadLocal` переменной одним виртуальным потоком, остаются в carrier-потоке (физическом потоке-носителе) после завершения виртуального потока. Затем другой виртуальный поток, выполняющийся на том же carrier-потоке, может некорректно увидеть эти "чужие" данные.
#### Причина

Виртуальные потоки не привязаны постоянно к одному физическому потоку. Они выполняются на carrier-потоках из общего пула (например, `ForkJoinPool`). Когда виртуальный поток блокируется на I/O или другой операции, он освобождает carrier-поток, который может быть переиспользован для другого виртуального потока.

`ThreadLocal` переменные хранятся в самом потоке (в поле `ThreadLocalMap`). Для обычных потоков это логично — данные живут вместе с потоком. Но для виртуальных потоков проблема в том, что:
- Виртуальный поток устанавливает значение в `ThreadLocal`.
- Затем он завершается или приостанавливается, но не очищает `ThreadLocal`.
- Carrier-поток, на котором выполнялся виртуальный поток, сохраняет это значение в своей `ThreadLocalMap`.
- Следующий виртуальный поток, выполняемый на этом же carrier, может прочитать это "левое" значение, если обратится к тому же `ThreadLocal`.

Это и есть **загрязнение** — данные одного виртуального потока "просачиваются" в другой.

#### Последствия

- **Утечка контекста**: например, идентификатор пользователя или транзакции может случайно использоваться в другом потоке, что приведёт к нарушению безопасности или логическим ошибкам.
    
- **Некорректное поведение**: код, полагающийся на `ThreadLocal` для хранения состояния (например, текущая сессия, соединение с БД), может работать непредсказуемо.
    
- **Трудности отладки**: ошибки проявляются нерегулярно, в зависимости от того, как виртуальные потоки распределяются по carrier-потокам.

### Синхронизация с synchronized

#### Особенности виртуальных потоков

Виртуальные потоки спроектированы так, чтобы **не блокировать carrier-поток** (физический поток ОС) при ожидании I/O или других операциях, которые могут занять много времени. Когда виртуальный поток встречает блокирующий вызов (например, чтение из сети), он **открепляется (unmount)** от своего carrier-потока, сохраняя своё состояние (стек, локальные переменные). Carrier-поток освобождается и может выполнять другие виртуальные потоки. После завершения блокирующей операции виртуальный поток снова **прикрепляется (mount)** к какому-либо свободному carrier-потоку и продолжает выполнение.

Это позволяет одному физическому потоку обслуживать тысячи виртуальных, эффективно утилизируя ресурсы.

#### Проблема synchronized: привязка к carrier-потоку

К сожалению, стандартная реализация `synchronized` в JVM не поддерживает открепление виртуального потока при ожидании монитора. Причина — в низкоуровневой реализации мониторов, которые тесно связаны с конкретным потоком ОС (carrier-потоком). Когда виртуальный поток входит в `synchronized` блок и монитор занят, он **не может открепиться** — вместо этого он блокирует **carrier-поток**, на котором выполняется.

Это приводит к тому, что carrier-поток простаивает в ожидании освобождения монитора, не имея возможности выполнять другие виртуальные потоки. Такая ситуация называется **pin-ning** (закрепление) виртуального потока за carrier-потоком.

#### Последствия pin-ning

1. **Снижение масштабируемости** – если много виртуальных потоков конкурируют за один монитор, они будут занимать carrier-потоки, даже если просто ждут. Это может привести к тому, что все доступные carrier-потоки окажутся заблокированными, и новые виртуальные потоки не смогут выполняться, хотя процессор простаивает.
    
2. **Дефицит carrier-потоков** – пул carrier-потоков (обычно `ForkJoinPool`) имеет ограниченный размер (по умолчанию равен числу ядер). Если каждый заблокированный виртуальный поток удерживает carrier, другие виртуальные потоки (даже не конкурирующие за этот монитор) могут остаться без выполнения.
    
3. **Возможный starvation** – длительное удержание монитора в сочетании с pin-ning может привести к голоданию других потоков.
