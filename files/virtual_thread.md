## Java 21: Виртуальные потоки

До появления виртуальных потоков в Java было два основных подхода к разработке:
- Блокирующий подход
- Асинхронный подход
## Блокирующий подход
Классика программирования. Spring MVC.
***Концепция: 1 запрос – 1 поток***

```kotlin
private fun selectUserFromDataBase(userId: Long): User {  
    val dataBaseRequest = createRequest(userId)  
    val rowSet = dataBaseClient.performRequest(dataBaseRequest)  
    val user = remapUser(rowSet)  
    return user  
}
```

| action                                         | time                                       | cpu                                   |
| ---------------------------------------------- | ------------------------------------------ | ------------------------------------- |
| createRequest(userId)                          | 0.000100 ms                                | CPU 100%                              |
| dataBaseClient.performRequest(dataBaseRequest) | <font style="color:red">100.0000 ms</font> | <font style="color:red">CPU 0%</font> |
| remapUser(rowSet)                              | 0.000100 ms                                | CPU 100%                              |
***Основная проблема - процессор простаивает 99% времени на операциях I/O***

## Основаная задача:
- Увеличить утилизацию процессора.
- Увеличить пропускную способность приложения.

И тут на сцену выходит

## Асинхронный подход

Реактивный стек, полностью асинхронный подход к программированию.
***Концепция: несколько запросов – 1 поток***
#### Плюсы:
- Улучшают пропускную способность приложения
- Эффективнее утилизирует CPU
#### Минусы:
- Много фреймворков: RxJava, Reactor, WebFlux, CompletableFuture
- Высокий порог входа
- Сложно читаемый код
- Сложности с отладкой
- Сложности с обработкой ошибок
- Сложности с тестированием

## Решение:  Создавать больше потоков

Почему такой подход не работает c платформенными потоками Java:
- Потоки в Java - это обертка над потоками ядра (потоками платформы)
- Создание нового потока требует ~2 MB оперативной памяти, создание, например, миллиона потоков потребует 2 TB оперативной памяти
- Создание нового потока требует времени, порядка нескольких миллисекунд, для создания миллиона потоков потребуется тысяча секунд(~15 минут)
- Если попробуем создавать потоки таким образом, то упремся в лимиты системы задолго до того, как создадим миллион потоков (1000-2000 потоков, в зависимости от железа)

## Решение:  Виртуальные потоки Java

***Концепция: 1 запрос – 1 поток***

### Преимущества виртуальных потоков:
- Виртуальные потоки - это новые потоки в мире Java
- Виртуальные потоки – легковесные, они недороги для создания
- Можно создать миллион потоков
-  Улучшают пропускную способность приложения
- Снижается потребление памяти приложением
- Больше не надо опираться на асинхронный неблокирующий код (асинхронный подход)
- Не нужно создавать пулы потоков, виртуальные потоки создаются по запросу - новый на каждый запрос, после чего уничтожаются
- Идеально подходят для блокирующего кода, такой как вызов REST, обращение к БД и т.д.
- Эффективнее утилизирует CPU
